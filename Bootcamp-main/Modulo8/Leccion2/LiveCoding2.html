01 — JERARQUÍA

Relación entre IA, ML y Deep Learning

Deep Learning no es sinónimo de IA ni de Machine Learning. Son conceptos anidados: el DL es un subconjunto del ML, que a su vez es un subconjunto de la Inteligencia Artificial.

Inteligencia Artificial Toda técnica que permite a las máquinas imitar comportamiento inteligente (incluye reglas, búsqueda, lógica, aprendizaje).
Machine Learning Subconjunto de IA donde los modelos aprenden patrones directamente desde datos, sin ser programados explícitamente.
Deep Learning Subconjunto de ML que usa redes neuronales profundas (múltiples capas) para aprender representaciones jerárquicas automáticamente.

La diferencia clave: en ML clásico el ingeniero extrae características manualmente (feature engineering). En DL, la red aprende esas representaciones sola, lo que lo hace especialmente poderoso con datos no estructurados y a gran escala.

02 — TIPOS DE DATOS

Estructurados vs No estructurados — ¿Cuándo usar DL?

La elección entre ML clásico y Deep Learning depende en gran medida del tipo de datos disponibles y el volumen de ejemplos de entrenamiento.

📊 Datos Estructurados

  • Tablas SQL / CSV
  • Series temporales financieras
  • Registros de sensores IoT
  • Datos de CRM / ERP
  • Formularios médicos
  • Resultados de encuestas

→ ML clásico (XGBoost, Random Forest) suele ser más eficiente aquí.

🧠 Datos No Estructurados

  • Imágenes y video
  • Audio y voz
  • Texto libre (NLP)
  • Gráfos y redes
  • Datos 3D / LiDAR
  • Secuencias biológicas (ADN)

→ Deep Learning domina. Las redes aprenden representaciones automáticamente.

✓ Usar DL cuando…

  • Tienes millones de ejemplos de entrenamiento
  • Los datos son imágenes, audio o texto
  • Las features son difíciles de definir manualmente
  • El problema es percepción, generación o traducción
  • Hay capacidad de cómputo (GPU/TPU)

✗ Preferir ML clásico cuando…

  • Los datos son escasos (<10K ejemplos)
  • Necesitas interpretabilidad y explicabilidad
  • Los datos ya están tabulados y bien procesados
  • Los recursos computacionales son limitados
  • La latencia de inferencia es crítica
03 — ARQUITECTURAS

DNN, CNN, RNN y Transformers

Cada arquitectura fue diseñada para capturar un tipo distinto de estructura en los datos. Seleccionar la correcta es fundamental para el rendimiento del modelo.

Deep Neural Networks (DNN)

Red totalmente conectada (fully-connected). Cada neurona de una capa se conecta a todas las de la siguiente. Es la arquitectura base del DL.

Datos tabulares Clasificación general Regresión

Ideal para datos tabulares con features ya definidas. No asume ninguna estructura espacial ni temporal en los datos. Limitado para imágenes o secuencias por su alto número de parámetros.

Convolutional Neural Networks (CNN)

Utiliza filtros convolucionales que detectan patrones locales (bordes, texturas, formas). Aprovecha la localidad espacial de imágenes; los parámetros se comparten entre posiciones.

Visión por computadora Clasificación de imágenes Detección de objetos Segmentación Audio (espectrogramas)

Arquitecturas icónicas: VGG, ResNet, EfficientNet, YOLO. Son el estándar en tareas de visión. La idea de "peso compartido" reduce dramáticamente los parámetros frente a DNN puras.

Recurrent Networks — RNN / LSTM / GRU

Redes con conexiones recurrentes: la salida de un paso se retroalimenta al siguiente. Diseñadas para datos secuenciales donde el orden importa. LSTM y GRU resuelven el problema de gradientes desvanecientes del RNN simple.

Series temporales NLP (pre-Transformers) Generación de texto Traducción automática Predicción financiera

Hoy han sido reemplazadas en NLP por Transformers, pero siguen siendo relevantes para series temporales de longitud moderada y donde el contexto secuencial local es clave.

Transformers & Mecanismo de Atención

Introducidos en "Attention is All You Need" (2017). En lugar de procesar secuencias paso a paso, atienden a todos los tokens simultáneamente vía self-attention. Escalables y paralelizables.

NLP / LLMs Vision Transformers (ViT) Multimodal Generación de imágenes Bioinformática (AlphaFold)

La arquitectura dominante actual. La base de GPT, BERT, T5, LLaMA, Stable Diffusion y prácticamente todos los modelos de frontera. Requiere grandes datasets y cómputo, pero Transfer Learning lo hace accesible.

ARQUITECTURA MEJOR PARA DEBILIDAD
DNN Datos tabulares y clasificación general No captura localidad ni orden
CNN Imágenes, audio, datos 2D No modela dependencias largas
RNN/LSTM Series temporales, secuencias cortas Lento, difícil de paralelizar
Transformer NLP, visión, multimodal, LLMs Costo computacional alto
04 — TRANSFERENCIA DE CONOCIMIENTO

Modelos Preentrenados y Transfer Learning

Entrenar una red profunda desde cero requiere millones de datos y semanas de cómputo. El aprendizaje por transferencia reutiliza el conocimiento de un modelo preentrenado en una tarea grande para adaptarlo a una tarea específica con pocos datos.

🌐
Modelo Base
Preentrenado en dataset enorme (ImageNet, Common Crawl)
→
❄️
Congelar Capas
Se mantienen los pesos de las capas bajas (features genéricas)
→
🔧
Fine-Tuning
Se re-entrena la "cabeza" del modelo con datos de la tarea nueva
→
🎯
Modelo Adaptado
Alto rendimiento con mucho menos datos y tiempo

¿Por qué funciona? Las capas iniciales de una red aprenden features universales (bordes, frecuencias, n-gramas) que sirven para casi cualquier problema dentro del mismo dominio. Solo las capas finales aprenden la tarea específica.

IMPACTO REAL

BERT redujo de semanas a horas el tiempo necesario para alcanzar SOTA en NLP. ResNet preentrenado permite clasificar imágenes médicas con solo ~500 ejemplos. GPT-4 fue fine-tuneado con ~10K ejemplos RLHF. Transfer Learning democratizó el DL.

05 — ECOSISTEMA

Frameworks: TensorFlow, Keras y PyTorch

Los tres frameworks dominan el ecosistema del DL. La elección depende del contexto: producción, investigación, prototipado rápido o despliegue en dispositivos.

TensorFlow
GOOGLE BRAIN · 2015
Mejor para: producción a escala, despliegue móvil y edge (TFLite), pipelines MLOps completos.
  • Ecosistema maduro (TFX, TFServing, TFLite)
  • Excelente soporte para TPUs
  • TensorBoard para visualización
  • Grafos estáticos = optimización en producción
Keras
API DE ALTO NIVEL · INTEGRADO EN TF
Mejor para: prototipado rápido, docencia, investigadores que priorizan velocidad de iteración.
  • API intuitiva y legible
  • Menos código boilerplate
  • Excelente para comenzar con DL
  • Compatible con múltiples backends (TF, JAX, PyTorch)
PyTorch
META AI · 2016
Mejor para: investigación, desarrollo de nuevas arquitecturas, comunidad académica, LLMs.
  • Grafos dinámicos (define-by-run)
  • Debugging intuitivo con Python nativo
  • Domina la investigación (papers en NeurIPS, ICML)
  • HuggingFace, Lightning, Triton, TorchServe

Regla práctica: si estás investigando o construyendo un prototipo, usa PyTorch. Si necesitas llevar a producción en Google Cloud o dispositivos móviles, TensorFlow/Keras. En la práctica, muchos equipos usan PyTorch para desarrollo y exportan a ONNX o TorchServe para producción.
